# Sentinel Security Architecture

Sentinel operates as an Enterprise-Grade, Governance-Driven Security Pipeline Engine. It is not merely a scanner, but an auditable decision engine built for CI/CD environments. The architecture separates untrusted code execution from forensic evaluation through a strict Layered Trust Model.

## The 5-Layer Security Pipeline

### Layer 1: Static Engine (Local First)
- **Role:** Deterministic parsing and AST extraction.
- **Constraints:** Parses source files without executing them. Uses regex and structural walking (`acquitern`) to match syntax patterns and identify risky primitives.
- **Ruleset:** Static signatures, hardcoded configuration files (e.g., .npmrc override rules), and transitive dependency maps.

### Layer 2: Adaptive Engine Scoring
- **Role:** Behavioral anomaly evaluation. 
- **Constraints:** Uses the `ConfidenceScorer` and `TriggerLevelOrchestrator`. Multiple seemingly low-risk signals are mathematically combined (Logistic Metric Evaluation) to detect advanced evasion techniques and polymorphic threats, triggering an incident only when the composite risk score breaches established noise caps. 

### Layer 3: Sandbox Verification (Cloud CI/CD)
- **Role:** Controlled runtime detonation.
- **Constraints:** Only activated under `--mode=sandbox`. Requires ephemeral execution contexts (GitHub Actions or Docker). The engine verifies malicious behavior through isolation profiling (Execution/Network traces) to definitively assert High Trust findings.

### Layer 4: Disconnected Visual Validation
- **Role:** Forensic evaluation by the user.
- **Constraints:** Sentinel uses a decoupled UI architecture using React. The Dashboard is "Local-First" and runs fully offline. It consumes standardized JSON outputs generated by Layer 1/2/3. It never parses or executes original raw inputs to protect the developer running the investigation. Validation uses a rigid "Fail-Closed" Zod Schema implementation to prevent buffer or render overloads.

### Layer 5: Operational Governance and Auditability
- **Role:** Compliance and Regulatory Release Enforcement.
- **Constraints:** 
  1. **Rulepack Versioning:** Every scan strictly encodes its baseline heuristic version (e.g., `rulepack_version: 2026.04`). This allows comparative security audits across time.
  2. **Audit Signatures:** Every heuristic trigger mandates a `rule_id` (e.g. `SARB-EXEC`) and a formal `explanation` of the exact pattern matched.
  3. **CI/CD Regression Gates:** Sentinel is fortified by automated Regression Suites (`scripts/benchmark_suite`) that execute in pre-merge blocks. A regression in threat detectability or an increase in False Positive Rates natively aborts the enterprise build via deterministic exit codes.

## Trust Metric Architecture

Findings generated by Sentinel emit a mathematical `confidence_score` assigned explicitly by the Engine (Source of Truth).
- **High Trust (>90%):** Threat validated in an Isolated Execution environment.
- **Medium Trust (<80%):** Threat mathematically derived purely from Static/Adaptive AST. Carries potential False Positives but necessitates operational investigation.
